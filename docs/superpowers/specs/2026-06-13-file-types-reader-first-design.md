# file-types.html 독자 중심 재구성 설계

작성일: 2026-06-13
대상 파일: `file-types.html`
분류: Reference 페이지 개선 / 목표 등급 9.5점

## 왜

비개발자 비즈니스 리더에게 이 화면을 띄워놓고 설명한다고 가정하면, 지금 페이지는 확장자 8종을 평평한 카탈로그로 나열한다. 독자는 "내가 왜 `.ts`나 `.sh`를 알아야 하지"라는 연결고리 없이 비교표를 마주한다.

핵심 진단:
1. 왜 알아야 하는지(도입 framing)가 없다.
2. 8종을 같은 무게로 다뤄 우선순위가 안 보인다. 실제로 리더가 직접 여는 건 `CLAUDE.md`(.md)와 `settings.json`(.json) 둘뿐이다.
3. 영문 풀네임(HyperText Markup Language 등)이 비교표 핵심 칸을 차지해 진입장벽이 된다.
4. 모든 파일마다 검은 코드블록이 붙어 비개발자에게 노이즈가 된다.
5. 강점: `section-scenario`("언제 만나는지")와 HTML=뼈대/CSS=외모/JS=동작 비유는 방향이 맞다. 키운다.

## 무엇 (확정 결정)

범위는 "독자 중심 재구성"(9.5점), 코드는 "유지 + 한 줄 해설".

### A. 도입 오리엔테이션 블록 (신규, 비교표 위)
3문장 길잡이.
- 확장자 정의: 파일 이름 끝, 점 뒤 글자가 파일 종류.
- 안심 장치: 8개가 나오지만 직접 여는 건 2개(`CLAUDE.md`, `settings.json`)뿐.
- 나머지는 Claude가 만들고 독자는 알아보기만.

### B. 비교표 개선
- "이름" 칸 영문 풀네임을 작은 보조 텍스트로 강등, 주인공은 쉬운 한 줄 의미.
- 각 행에 `✏️ 직접 편집` / `👀 읽기만` 관계 배지 추가.
- 정렬: 직접 편집(.md, .json)이 위로.

### C. 섹션 순서 재배열 (우선순위순)
비교표 → 마크다운(.md) → 설정·데이터(.json·.ts) → 웹(.html·.css·.js) → 자동화(.py·.sh)
- 직접 여는 2개(.md, .json)를 앞에 모은다.
- 마크다운은 "AI가 선호하는 이유" 3카드를 가진 강한 섹션이라 1번 히어로 유지.
- 각 섹션 헤더에 `✏️`/`👀` 배지.
- `.ts`는 "JS의 안전 버전" 정도로 비중 축소.
- sub-menu도 4섹션으로 라벨/순서 갱신.

### D. 코드 스니펫 + 한 줄 해설
각 코드블록 바로 아래 `이게 무슨 뜻이냐면:` 평서문 한 줄.

### E. 비유 통일
웹 3총사를 집 짓기 비유로 묶음: HTML=골조, CSS=인테리어, JS=전기·동작.

## 제약 (기존 규약)
- 본문 em dash 절대 금지, 자연스러운 한국어.
- 새 박스에 왼쪽 컬러 테두리 클리셰 금지(색은 라벨 배지에). 기존 `section-scenario` 패턴은 재사용 가능.
