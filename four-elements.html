Why · 왜 4가지 도구인가

월요일 오전 9시, 김 상무의 책상

결재 대기 중인 부서 보고서가 7건 올라와 있습니다. 거래처와의 계약서 3건은 오늘 안에 최종 검토를 끝내야 하고, 오후 2시 임원 회의에 올릴 주간 보고서는 아직 정리되지 않았습니다. 점심 전에 의사결정 회신 12건도 마쳐야 합니다.

📄 결재 대기 7건 📑 계약서 최종 검토 3건 📊 임원 회의 자료 📧 의사결정 회신 12건

팀원에게 다시 시킬 시간도 없습니다. 새로운 팀이 필요합니다. 사람을 더 뽑을 수는 없습니다. 그래서 김 상무는 AI로 팀을 꾸리기로 했습니다.

그 팀을 만드는 4가지 도구가 에이전트 · 서브에이전트 · 스킬 · 후크입니다.

김 상무의 하루, Before와 After

같은 인물이 4가지 도구를 차례로 도입하며 겪는 변화입니다

Before · 도구 없이
📊
임원 회의 자료, 매주 두 시간
부서에서 올라온 보고서 7건을 직접 읽고, 핵심 수치만 추려 임원 회의용 한 장 요약본으로 정리합니다. 한 사람이 다 보기에는 양이 많습니다.
⏱ 약 2시간 / 회
After · 에이전트 + 서브에이전트 + 스킬
📊
한 줄 지시로 회의 자료 완성
에이전트가 부서별 보고서를 모읍니다. 서브에이전트가 항목별로 핵심 수치·이슈를 동시에 추립니다. 스킬이 임원 회의 양식에 맞춰 한 장으로 정리합니다. 김 상무는 결정만 합니다.
⏱ 약 10분 / 회 (임원 회의 자료 1회 기준, 사내 PoC 평균 n=12)
에이전트 서브에이전트 스킬
Before · 도구 없이
🔍
계약서 결재, 한 건당 40분
조항·금액·날짜·특약을 한 줄씩 눈으로 비교합니다. 결재선에 올라온 3건이면 두 시간, 놓치면 다시 돌려보내야 합니다.
⏱ 약 2시간 / 3건
After · 스킬 + 서브에이전트 + 후크
🔍

Agent 비서 실장

① 에이전트 (Agent)

알아서 일을 처리하는 비서 실장

개념 한 줄

한 줄 지시를 받으면 파일을 열고, 검색하고, 실행까지 직접 합니다. 질문에 답만 하는 챗봇이 아니라, 손이 있는 비서입니다.

📌 김 상무가 쓰는 법
지시 한 마디

"이 폴더의 회의록을 읽고 안건 3가지로 요약해줘"

↓
받는 결과

안건이 정리된 요약 파일 1개. 검토만 하면 됩니다.

👔 위임할 때 정할 것
  • 권한 범위 한 폴더에서만 읽기·쓰기. 처음부터 전체 폴더를 열어주지 않습니다.
  • 멈출 신호 결과 파일이 만들어지면 자동 종료. 외부 전송은 사람이 확정합니다.
참고: 직접 시도해보는 법
  1. 터미널에서 claude 입력해 Claude Code 실행
  2. "이 폴더에 어떤 파일이 있는지 알려줘"라고 입력
  3. 에이전트가 직접 파일 목록을 탐색하고 정리하는 과정을 확인
Sub-Agent 여러 전문가 팀

② 서브에이전트 (Sub-Agent)

비서 실장이 부르는 분야별 전문가 팀

개념 한 줄

비서 실장(에이전트) 혼자 다 하면 느립니다. 서브에이전트는 분야별 전문가를 동시에 불러 일을 나누는 구조입니다. 탐색·분석·검수가 동시에 돌아갑니다.

📌 김 상무가 쓰는 법
지시 한 마디

"이 폴더의 보고서에서 수치 오류·맞춤법·형식 불일치를 동시에 검사해줘"

↓
받는 결과

3개 항목이 동시에 점검된 결과 리포트. 한 줄짜리 지시로 세 사람 분 작업이 끝납니다.

👔 위임할 때 정할 것
  • 검수자 1명 결과를 모아 합치는 사람을 반드시 정합니다. 빠른 만큼 어긋나기 쉽습니다.
  • 분야 분리 서로 겹치지 않게 나눕니다. 같은 일을 두 명이 하면 비용만 늘어납니다.
참고: 전문가 4종 표 / 직접 시도
전문가하는 일언제 부를까?
🔍 탐색폴더에서 파일·내용 검색원하는 파일을 찾을 때
📋 기획작업 계획·전략 수립복잡한 작업을 시작할 때
⚡ 실행정리·변환 등 직접 실행실제로 손을 댈 때
🧩 종합여러 단계 조사·분석폭넓은 조사가 필요할 때
  1. 문서가 여러 개 있는 폴더를 엽니다
  2. "이 폴더의 파일들을 분석하고, 각 문서가 무슨 내용인지 동시에 조사해줘"라고 입력
  3. 여러 서브에이전트가 병렬로 실행되는 모습을 확인합니다
Skill 전문가들이 쓰는 공용 도구

③ 스킬 (Skill)

전문가들이 공유하는 표준 작업서

개념 한 줄

매번 같은 지시를 반복할 필요 없습니다. 스킬은 자주 쓰는 작업을 한 번 저장해두고, 짧은 명령어로 호출하는 매크로입니다. 엑셀 매크로, 카톡 빠른 답장과 같은 개념입니다.

📌 김 상무가 쓰는 법
한 번 만들어둠

실무자에게 "주간 보고서 검토 절차를 스킬로 만들어 공용 폴더에 올려라"고 지시

↓
매주 한 줄로 호출

/project:review 3월보고서 한 줄로 같은 품질의 검토가 매번 실행됩니다.

👔 위임할 때 정할 것
  • 표준화 안건 팀 회의에서 "스킬로 굳힐 작업 3개"를 합의합니다. 누가 만들든 결과가 같아야 합니다.
  • 변경 권한 공용 스킬을 누가 수정할 수 있는지 정합니다. 무단 수정은 품질 사고로 이어집니다.
참고: 동작 원리 / 저장 위치 / 스킬 파일 예시
이렇게 동작합니다
  • ① 미리 지시서를 텍스트 파일로 저장
  • ② 실행할 때 /project:리뷰 같은 짧은 명령어 입력
  • ③ Claude가 저장된 지시서대로 자동 실행
구분저장 위치대상실행
팀 공용.claude/commands/팀 전체/project:이름
개인~/.claude/commands/나만/user:이름
스킬 파일 예시 (review.md)
보고서 검토를 수행해주세요. - 수치 정확성 확인 - 맞춤법·문법 체크 - 양식 규칙 준수 여부 대상: (실행 시 추가 입력)
Hook 특정 상황에서 자동 실행

④ 후크 (Hook)

특정 상황이 되면 자동으로 울리는 알람

개념 한 줄

"이 상황이 되면 자동으로 이것을 실행해"라고 미리 정해두는 규칙입니다. 사람이 개입하지 않아도 시스템이 스스로 다음 단계를 실행합니다.

📌 김 상무가 쓰는 법
한 번 설정함

"보고서가 올라올 때마다 맞춤법·수치 자동 검사 후 결과를 메신저로 보내줘"

↓
매번 자동 실행

이후 보고서 업로드마다 사람 없이 검사가 돌고, 이상치만 메신저로 전달됩니다. 김 상무는 알림이 올 때만 보면 됩니다.

👔 위임할 때 정할 것
  • 최소 권한 처음엔 읽기 전용으로 시작합니다. 쓰기·외부 전송은 사후 승급합니다.
  • 롤백 경로 후크가 잘못 돌면 즉시 끌 수 있는 사람·방법을 미리 정합니다.
🛡️ 거버넌스 체크포인트
후크 도입 전 조직장이 점검할 4가지
권한 범위
어떤 폴더·명령까지 허용할지 명시. 기본은 읽기 전용.
민감 데이터
고객·계약·인사 데이터가 후크 출력에 노출되지 않는지 점검.
감사 로그
언제·무엇이 실행됐는지 기록을 남기고 검토자 지정.
롤백 절차
잘못된 동작 시 즉시 비활성화·복구할 경로 사전 마련.
참고: 4가지 발동 시점 / 설정 파일 예시
언제설명활용 예
🔒 시작 전도구 실행 전 먼저 실행위험 명령 자동 차단
✅ 완료 후파일 수정 직후 실행문서 자동 정리·검사
🔔 알림 시Claude 알림 전송 시Slack/이메일 전달
🏁 종료 시전체 작업 종료 시결과 보고서 자동 생성

아래 코드는 참고용입니다. Claude에게 "파일 수정 후 알림 후크 만들어줘"라고 말하면 자동 생성됩니다.

설정 파일 (.claude/settings.json)
{ "hooks": { "afterToolUse": [{ "matcher": "Edit|Write", "command": "echo '파일이 수정되었습니다'" }] } }
🚫 이럴 땐 자동화하지 마세요
조직장이 먼저 막아야 할 3가지 상황입니다
  1. 1회성 작업은 자동화하지 않습니다. 한 번 쓰고 버릴 작업을 위한 후크·스킬은 유지비용만 남깁니다.
  2. 판단 기준이 매번 바뀌는 작업은 자동화하지 않습니다. 사람이 매번 다르게 결정해야 한다면 그 결정 자체가 일입니다.
  3. 실패 시 책임 소재가 모호한 작업은 자동화하지 않습니다. 누가 책임지는지 정해지지 않으면 자동화가 사고를 가립니다.
자동화 금지가 신뢰의 시작입니다
📋 팀 도입 3단계 체크리스트
조직장이 챙겨야 할 도입 순서
1
파일럿 1~2명
반복 작업이 많은 실무자 1~2명에게 먼저 시도하게 합니다. 2주 안에 결과를 공유합니다.
2
공용 스킬 합의
팀 회의에서 표준화할 작업 3개를 정하고 .claude/commands/ 폴더에 공용 스킬로 올립니다.
3
후크 거버넌스
권한 범위·민감 데이터·감사 로그·롤백 룰셋을 문서화한 뒤에야 후크를 활성화합니다.

이제 김 상무의 4가지 도구를 당신의 팀에 조립할 차례입니다

팀 구조 설계(Solo → Orchestra)를 배웠고, 4가지 도구(에이전트·서브에이전트·스킬·후크)를 익혔습니다.
가장 먼저 만들 것은 팀이 매주 반복하는 작업 1개를 스킬로 굳히는 일입니다.
