다이내믹 워크플로우

작업을 받으면 Claude Code가 그 자리에서 하네스를 직접 짜서 실행합니다

개념

무엇인가: 복잡한 일을 Claude가 "팀으로 나눠" 처리하는 방식

보통 Claude에게 일을 시키면 한 명이 처음부터 끝까지 혼자 합니다. 다이내믹 워크플로우는 다릅니다. 복잡한 일을 받으면 Claude가 먼저 "이 일을 어떻게 쪼개서, 누구에게, 어떤 순서로 맡길지"를 계획하고, 여러 명의 Claude(서브에이전트)에게 나눠 맡깁니다. 이때 짜는 일 처리 계획과 역할 분담을 하네스라고 부릅니다.

한 문장으로

다이내믹 워크플로우는 작업을 받을 때마다 Claude가 그 일에 맞는 팀과 처리 순서(하네스)를 직접 짜서 돌리는 것입니다.

보통의 Claude 사용

  • 한 명이 혼자 처음부터 끝까지
  • 길고 복잡하면 중간에 놓치거나 대충 마무리
  • 막히면 사람이 단계를 다시 지시

다이내믹 워크플로우

  • 일을 쪼개 여러 명에게 분담
  • 만드는 쪽과 검증하는 쪽을 따로
  • 계획·분담·검증까지 Claude가 알아서

어떻게 나누나: Claude가 쓰는 네 가지 방식

작업에 맞춰 Claude가 알아서 골라 씁니다. 당신이 코드를 쓸 일은 없습니다

🤖

한 명에게 맡기기

서브에이전트 1명 · agent()

Claude 한 명을 띄워 일을 시키고 결과를 받습니다. 가장 기본 단위입니다.

🔀

동시에 여러 명

나눠서 한꺼번에 · parallel()

일을 쪼개 여러 명에게 동시에 맡기고 결과를 모읍니다. 그만큼 빨라집니다.

⛓️

단계로 넘기기

컨베이어처럼 · pipeline()

한 명의 결과를 다음 사람에게 넘기며 단계를 통과시킵니다. 초안 다음 검토 다음 정리처럼 흐릅니다.

🔁

끝까지 반복

기준을 채울 때까지 · loop

목표를 채울 때까지 같은 흐름을 되풀이합니다. 더 나올 게 없으면 멈춥니다.

용어 정리

하네스는 일 처리 계획과 역할 분담, 워크플로우는 그 작업용으로 새로 짠 하네스 한 벌입니다. 그래서 다이내믹 워크플로우는 "작업마다 새로 짜는 하네스"입니다.

왜 나누나: 혼자선 긴 작업에서 무너진다

혼자 끝까지 가려는 AI는 긴 작업에서 세 가지 방식으로 무너집니다. 그래서 만드는 일과 검증하는 일을 나누고, 더 나올 게 없을 때까지 반복하게 만듭니다.

😴

에이전트 게으름

일을 끝까지 하지 않고 절반쯤에서 "다 했다"며 멈춥니다.

Agentic laziness · 10개를 시켰는데 3개만 하고 마무리
🪞

자기 편향

다른 가능성보다 자기가 먼저 낸 결론을 계속 옳다고 여깁니다.

Self-preferential bias · 자기 답을 자기가 후하게 평가
🎯

목표 표류

요약을 거듭하는 사이에 처음 받은 목표에서 조금씩 멀어집니다.

Goal drift · 긴 작업 끝에 다른 일을 하고 있음

세 가지 모두 혼자 끝까지 가려는 한계입니다.
나누고, 검증하게 하고, 목표를 문서로 고정하면 풀립니다.

출처: Anthropic, "A harness for every task: dynamic workflows in Claude Code" · 원문

6가지 패턴
어떻게 푸나 Claude가 작업에 맞춰 골라 쓰는 여섯 가지 하네스 패턴. 각 패턴을 도식과 함께 정리했습니다
1분류 후 실행 (Classify-and-act)
언제: 들어온 일의 종류가 제각각일 때

먼저 작업이 어떤 종류인지 가른 다음, 종류에 맞는 처리로 보냅니다. 문의 티켓을 분류해 담당 창구로 넘기는 방식입니다.

2펼치고 합치기 (Fan-out-and-synthesize)
언제: 독립된 조각으로 나눌 수 있을 때

일을 여러 조각으로 펼쳐 동시에 처리하고, 결과를 하나로 모읍니다. 가장 자주 쓰는 기본 패턴입니다.

3적대적 검증 (Adversarial verification)
언제: 결과가 틀리면 안 되는 일일 때

만든 에이전트와 별개로, 결과를 깨뜨리려고 달려드는 검증 에이전트를 둡니다. 그래도 통과하면 신뢰도가 올라갑니다.

4생성 후 거르기 (Generate-and-filter)
언제: 후보를 많이 내고 좋은 것만 남길 때

여러 안을 일부러 많이 만든 뒤, 품질 기준으로 걸러 남깁니다. 양으로 시작해 질로 좁힙니다.

5토너먼트 (Tournament)
언제: 가장 나은 하나를 골라야 할 때

여러 결과를 둘씩 맞붙여 이긴 쪽을 올립니다. 비교를 거듭해 최종 한 개(Winner)를 가립니다.

6끝까지 반복 (Loop until done)
언제: 몇 번 걸릴지 모르는 일일 때

멈출 조건을 정해 두고, 그 조건을 채울 때까지 같은 흐름을 반복합니다. 더 나올 게 없을 때 끝냅니다.

패턴은 섞어 쓴다

실제 작업에서는 한 패턴만 쓰지 않습니다. 펼쳐서 동시에 처리하고(2), 각 결과를 적대적으로 검증하고(3), 더 나올 게 없을 때까지 반복하는(6) 식으로 겹쳐 씁니다.

한 작업이 실제로 도는 모습

예시 작업 하나가 앞의 패턴들을 거쳐 처리되는 과정입니다

예시 작업: "경쟁사 40곳의 최근 동향을 조사해 한 장으로 정리해줘"

  1. 1
    나눠서 동시에 조사 펼치고 합치기
    40곳을 여러 묶음으로 쪼개, 서브에이전트가 각 묶음을 병렬로 조사합니다.
  2. 2
    사실 검증 적대적 검증
    다른 에이전트가 각 요약의 출처와 수치를 의심하며 점검합니다.
  3. 3
    빠진 곳 보완 끝까지 반복
    검증을 통과하지 못한 곳만 다시 조사합니다. 더 나올 게 없을 때까지 되풀이합니다.
  4. 4
    한 장으로 합치기 펼치고 합치기
    검증된 내용만 모아 최종 보고서 한 장으로 정리합니다.
코드 작업도 똑같다

"이 저장소에서 안 쓰는 함수를 제거해줘"라면, 파일을 나눠 동시에 찾고(펼치고 합치기), 정말 안 쓰는지 다른 에이전트가 검증하고(적대적 검증), 빠짐없이 반복합니다(끝까지 반복). 작업만 바뀔 뿐 흐름은 같습니다.

실전 사용법

언제 쓰고, 언제 쓰지 않나

워크플로우는 토큰을 많이 씁니다. 가치가 큰 작업에만 씁니다

쓰지 않는 게 낫다

  • 한 번에 끝나는 일상 코딩
  • 단순 질문과 짧은 수정
  • 대화 한 번으로 충분한 작업

쓰면 좋다

  • 대규모 코드 마이그레이션과 리팩터
  • 여러 출처를 검증하는 딥리서치
  • 대량 데이터 정렬·분류·심사
  • 근본 원인 분석, 디자인 탐색

어떻게 부르나

워크플로우는 토큰을 많이 쓰는 기능이라 Claude가 함부로 켜지 않습니다. 그래서 "이 작업은 워크플로우로 풀어줘"라고 당신이 분명히 신호를 줘야 시작합니다. 신호를 받으면 Claude가 작업을 보고 하네스를 직접 설계해 실행합니다. 따로 설정하거나 코드를 쓸 일은 없습니다.

실전 프롬프트 
# 이 작업을 워크플로우로 풀어줘

작업: (여기에 큰 작업을 적는다)
  예) 이 저장소 전체에서 더 이상 쓰지 않는 함수를 찾아 안전하게 제거해줘

# 요구
  · 먼저 작업을 독립된 조각으로 나눠라 (펼치고 합치기)
  · 각 조각은 서브에이전트로 동시에 처리해라
  · 결과는 별도의 검증 에이전트가 적대적으로 점검해라
  · 더 나올 게 없을 때까지 반복하고, 끝나면 한 번에 합쳐서 보고해라

토큰을 많이 써도 좋으니 끝까지, 빠짐없이.
🔁

/loop과 함께

계속 돌리기

워크플로우를 주기나 조건에 맞춰 반복 실행해, 한 번에 끝나지 않는 일을 끝까지 끌고 갑니다.

🎯

/goal과 함께

목표 고정

목표를 문서로 박아 두면 긴 작업에서도 목표 표류가 줄어듭니다.

한 번 잘 만든 워크플로우는 자산이 된다

쓸 만한 워크플로우가 나오면 버리지 말고 남깁니다. ~/.claude/workflows 폴더에 저장하거나 스킬로 묶어 두면, 다음에 같은 종류의 작업이 올 때 그대로 다시 부를 수 있습니다. 한 번의 설계가 반복 가능한 도구로 바뀝니다.

주의

워크플로우는 여러 에이전트를 동시에 띄우는 만큼 토큰을 많이 씁니다. 모든 작업에 쓰지 말고, 결과의 가치가 비용을 넘는 작업에만 쓰세요.
