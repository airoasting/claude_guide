? MIT vs Apache 2.0, 한 줄로 말하면? ▾

MIT는 "그냥 맘대로 써. 내 이름만 남겨줘"이고,
Apache 2.0은 "맘대로 써. 대신 내 이름 남기고, 특허로 뒤통수치지 마"입니다.

둘 다 상업적 이용, 수정, 재배포, 비공개 모두 허용하는 허용적 라이선스입니다. 진짜 차이는 딱 3가지입니다.

  • ① 특허 조항의 유무 (가장 중요)
    Apache 2.0에는 "기여자가 자기 특허를 이 코드 사용자들에게 자동으로 허락한다"는 조항이 있습니다. 반대로 사용자가 이 코드를 가지고 기여자에게 특허 소송을 걸면 라이선스가 즉시 종료됩니다. MIT에는 이런 조항이 아예 없어서, 누군가 "이 코드 알고리즘은 내 특허야"라며 나중에 소송을 걸 수 있습니다.
    → 독창적인 알고리즘이나 기업 제품이면 Apache 2.0이 안전합니다.
  • ② 변경 사항 표시 의무
    Apache 2.0은 코드를 수정해서 재배포할 때 "어디를 바꿨는지 표시"를 요구합니다. MIT는 그런 요구가 없어서 그냥 수정해서 쓰면 됩니다.
  • ③ NOTICE 파일
    Apache 프로젝트를 가져다 쓸 때 원본에 NOTICE 파일이 있으면 반드시 함께 배포해야 합니다. MIT는 LICENSE 파일 하나만 챙기면 끝입니다.

요약: MIT는 텍스트 1장으로 끝나는 가장 단순한 라이선스, Apache 2.0은 거기에 "특허 분쟁 방지" 장치를 추가한 버전이라고 생각하면 됩니다. 개인 스킬은 MIT, 기업/특허 이슈가 있으면 Apache 2.0.

! 처음이라면 MIT, 특허가 걱정되면 Apache 2.0 ▾

개인 프로젝트, Claude Code 스킬, MCP 서버 등 대부분의 경우 MIT면 충분합니다. 조건이 가장 단순하고, 누구나 부담 없이 가져다 쓸 수 있어서 채택률이 가장 높습니다.

만약 내 스킬에 독창적인 알고리즘이나 처리 방식이 포함되어 있거나, 기업 환경에서 배포할 예정이라면 Apache 2.0을 선택하세요. 특허 라이선스 조항이 포함되어 있어서, 누군가 내 코드를 가져다 쓴 뒤 역으로 특허 소송을 거는 상황을 방지할 수 있습니다.

잘 모르겠으면 MIT로 시작하고, 나중에 필요할 때 바꿔도 됩니다.

! MIT/Apache 라이선스 파일을 사용할 때 주의할 점 ▾
  • LICENSE 파일을 반드시 포함하세요. npm 패키지든 GitHub 저장소든, 루트 디렉토리에 LICENSE 파일이 없으면 기본 저작권법이 적용되어 아무도 합법적으로 사용할 수 없습니다.
  • 저작권 표시의 연도와 이름을 정확히 기입하세요. [작성자 이름]을 그대로 두면 법적 효력이 없습니다. 본인 이름 또는 조직명을 넣어야 합니다.
  • 다른 오픈소스를 가져다 쓸 때는 원본의 LICENSE도 함께 포함하세요. MIT/Apache 모두 "원본 저작권 표시를 유지할 것"을 요구합니다. 의존성 라이브러리의 라이선스 파일을 삭제하면 위반입니다.
  • Apache 2.0의 NOTICE 파일을 무시하지 마세요. Apache 프로젝트에 NOTICE 파일이 있다면 배포 시 반드시 포함해야 합니다. 많은 사람이 놓치는 부분입니다.
  • MIT와 Apache는 섞어서 사용할 수 있습니다. 내 스킬이 MIT이고 사용하는 라이브러리가 Apache 2.0이어도 문제없습니다. 다만, GPL 코드를 MIT 프로젝트에 포함하면 전체가 GPL이 되니 주의하세요.
  • "AS IS" 면책 조항을 이해하세요. 두 라이선스 모두 "이 소프트웨어는 있는 그대로 제공되며, 어떠한 보증도 하지 않는다"는 면책 조항을 포함합니다. 내 스킬 때문에 문제가 생겨도 법적 책임을 지지 않는다는 의미입니다.

가장 많이 선택되는 스킬/도구 라이선스

이름의 유래: 미국 매사추세츠 공과대학교(Massachusetts Institute of Technology)에서 1980년대에 X 윈도 시스템을 배포하며 처음 사용한 라이선스입니다.

내가 만든 Claude Code 스킬을 최대한 널리 퍼뜨리고 싶을 때 최적의 선택입니다. 사용자는 저작권 표시만 남기면 자유롭게 수정, 재배포, 상업적 이용이 가능합니다. npm 패키지, GitHub 공개 저장소 등 대부분의 오픈소스 도구가 이 라이선스를 사용합니다.

특허 보호가 필요한 기업용 스킬에 적합

이름의 유래: 1995년에 시작된 Apache HTTP Server 프로젝트에서 만든 라이선스입니다. Apache라는 이름은 초기 개발자들이 기존 서버 코드에 패치(patch)를 많이 적용해서 "A Patchy Server"라는 말장난에서 유래했습니다.

MIT만큼 허용적이지만 특허 라이선스 조항이 포함되어 있습니다. 내 스킬에 독창적인 알고리즘이나 처리 방식이 있고, 누군가 그걸 특허로 걸어서 역으로 나를 고소하는 상황을 막고 싶다면 Apache 2.0이 안전합니다.

✓ 상업적 이용 ✓ 수정 ✓ 배포 ✓ 특허 라이선스 ⚠ 저작권 표시 ⚠ 변경 사항 명시 ⚠ NOTICE 파일 보존
스킬 배포자 관점
  • 사용자가 내 스킬을 수정하면 변경 사항을 표시해야 함
  • 기여자가 관련 특허를 보유해도 자동으로 특허 사용 허락
  • 특허 소송을 걸면 라이선스가 자동 종료 → 악의적 소송 방지
  • NOTICE 파일로 기여자/출처 정보를 체계적으로 관리

카피레프트(Copyleft) 라이선스
GPL
GPL v3.0
GNU General Public License v3.0

이름의 유래: 리처드 스톨먼(Richard Stallman)이 1989년에 자유 소프트웨어 운동을 위해 만든 라이선스입니다. GNU는 "GNU's Not Unix"의 재귀 약자이며, General Public License는 "모든 사람을 위한 공개 라이선스"라는 의미입니다.

가장 강력한 카피레프트 라이선스입니다. 내 스킬에 GPL 코드를 포함해서 배포하면 전체 소스코드를 동일한 GPL 라이선스로 공개해야 합니다.

✓ 상업적 이용 ✓ 수정 ✓ 배포 ⚠ 소스코드 공개 ⚠ 동일 라이선스 적용 ⚠ 변경 사항 명시 ✗ 독점 소프트웨어 결합 제한
핵심 요약
  • 파생 저작물도 반드시 GPL로 배포해야 함 (전염성)
  • 바이너리 배포 시 소스코드를 함께 제공하거나 접근 방법 안내 필수
  • v3는 특허 보호와 DRM 우회 금지 방지 조항 포함
  • SaaS(서버에서만 실행)는 배포에 해당하지 않아 공개 의무 없음

LGL
LGPL v3.0
GNU Lesser General Public License v3.0

이름의 유래: GPL에서 파생된 라이선스로, Lesser(더 작은)라는 이름은 GPL보다 제약이 약하다는 뜻입니다. 원래는 Library GPL이라고 불렸지만, "라이브러리에만 쓰라"는 오해를 막기 위해 Lesser로 이름을 바꿨습니다.

GPL의 완화된 버전입니다. 내 스킬이 LGPL 라이브러리를 그대로 가져다 연결해서 쓰는 경우에는 내 코드에 다른 라이선스를 적용할 수 있습니다. 단, 라이브러리 자체를 수정하면 수정한 부분은 LGPL로 공개해야 합니다.

✓ 상업적 이용 ✓ 라이브러리 링크 자유 ⚠ 라이브러리 수정 시 소스 공개 ⚠ 저작권 표시
핵심 요약
  • 라이브러리를 그대로 사용(링크)하면 자체 코드 공개 불필요
  • LGPL 라이브러리를 수정하면 수정된 부분만 LGPL로 공개
  • 독점 소프트웨어에서도 LGPL 라이브러리 사용 가능
대표 프로젝트: GNU C Library, Qt(일부), FFmpeg(일부)

이름의 유래: Affero라는 이름은 이 라이선스를 처음 만든 회사 Affero, Inc.에서 왔습니다. Affero는 라틴어로 "가져오다, 전달하다"라는 뜻으로, 네트워크를 통해 소프트웨어를 전달하는 상황까지 라이선스를 확장하겠다는 의도가 담겨 있습니다.

GPL의 강화 버전입니다. 내 스킬을 서버에서 실행하는 것만으로도 '배포'로 간주하기 때문에, 네트워크를 통해 서비스를 제공하는 경우에도 소스코드를 공개해야 합니다.

✓ 상업적 이용 ⚠ 소스코드 공개 ⚠ 네트워크 서비스도 공개 대상 ⚠ 동일 라이선스 적용 ✗ SaaS 루프홀 차단
핵심 요약

  • 서버에서 AGPL 코드를 실행하면 사용자에게 소스 접근 방법 제공 필수
  • SaaS 기업에서 가장 주의해야 하는 라이선스
