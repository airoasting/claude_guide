루프 엔지니어링이란

사람이 매번 붙어 있지 않아도, AI가 목표까지 스스로 끝내게 만드는 일입니다.

예를 들어 밤사이, 실패한 테스트가 0건이 될 때까지, 또는 분기 보고서가 검토 항목을 전부 통과할 때까지 AI가 알아서 고치고 다시 돌려 둡니다. 아침에는 결과만 확인하면 됩니다.

실행자 · 매번 지시→감독관 · 목표와 기준만 정함

AI를 다루는 방식은 프롬프트에서 세 번 진화했습니다.
말 한마디를 잘 쓰는 일에서, 작업 환경을 갖추는 일로, 운영 틀을 두르는 일로, 다시 도는 시스템을 짜는 일로.

한 줄 정의

루프 엔지니어링은 하네스 한 틀을 목표에 닿을 때까지 반복해서 돌리는 일입니다. 사람은 이 반복 구조를 설계하고 운영하며, 결과를 보면서 계속 다듬습니다. 이때 사람의 역할은 매번 지시하는 실행자에서 루프를 지켜보는 감독관으로 옮겨갑니다.

Trigger

Trigger는 루프를 언제 시작할지 정합니다

루프 엔지니어링에서 실제로 설계할 것은 세 가지입니다. 목표에 닿을 때까지 돌릴지, 정해진 시간마다 돌릴지, 반복 업무 전체를 자동 운영 구조로 넘길지 먼저 정합니다.

완료 기준이 분명한 일은 Goal-based로 시작합니다. 일이 정해진 간격으로 들어오는 경우는 Time-based로 옮깁니다. 반복 작업이 충분히 안정된 뒤에 Proactive로 키웁니다.

1Goal-based loop· 목표 도달형
멈춤 조건을 정함

핵심: 완료 기준과 최대 반복 수를 먼저 둡니다. Claude가 멈추려 할 때 평가 모델이 조건을 보고, 부족하면 다시 일하게 합니다.

/goal 템플릿 
/goal 첨부한 제안서를 아래 검토 기준을 모두 통과할 때까지 다듬어 주세요. 5번 고쳐도 통과하지 못하면 멈추고, 남은 문제를 정리해 알려 주세요.

# 완료 조건 (모두 충족되면 끝내 주세요)
  · 첫 문단만 읽어도 핵심 제안과 기대 효과가 드러날 것
  · 모든 수치와 인용이 원본 자료와 100% 일치할 것
  · 결론이 앞의 근거에서 자연스럽게 이어질 것

# 마무리
  · 무엇을 어떻게 고쳤는지 요약과 함께 최종본을 보고해 주세요
2Time-based loop· 정기 실행형
시작 조건을 정함

핵심: 사람은 매번 부르지 않고 시간표를 넘깁니다. Claude는 정해진 간격에 외부 변화를 확인하고, 변화가 있으면 처리합니다.

/loop 템플릿 
/loop 5m 공유 메일함에 새로 들어온 고객 문의를 확인하고, 답변 초안을 준비해 주세요.

# 멈출 조건 (충족되면 마쳐 주세요)
  · 처리할 새 문의가 없으면 멈춰 주세요
  · 같은 문의가 3번 반복되면 멈추고, 원인을 짚어 알려 주세요

# 매 실행 (한 바퀴마다 해 주세요)
  · 새로 온 문의만 골라 유형별로 분류해 주세요
  · 답변 초안을 작성하되, 발송은 하지 말고 대기해 주세요
  · 이번 바퀴에 처리한 내용을 한 줄로 기록해 주세요
3Proactive loop· 자동 위임형
맡기고 승인만 함

핵심: 사람이 실시간으로 붙어 있지 않습니다. 스케줄, 목표, 검증 Skill, 동적 워크플로우, 자동 모드를 조합하고 사람은 최종 승인 지점에 섭니다.

Proactive 템플릿 
/schedule every day 08:30 지난 하루의 업계 소식을 모아 아침 브리핑 초안을 만들어 주세요.

# 목표
  · 우리 사업과 관련된 소식만 중요도 순으로 골라 주세요
  · 각 항목을 사실과 시사점으로 나눠 정리해 주세요

# 작업 흐름
  · 초안을 만든 뒤, 검토 단계에서 사실과 출처를 다시 확인해 주세요
  · 검토를 통과하면 발송 대기 상태로 정리해 주세요
  · 실제 발송은 제 승인을 받은 뒤에 진행해 주세요
Stop

Stop은 루프가 언제 멈출지 정합니다

루프는 멈출 조건이 분명해야 안전하게 돕니다. 성공 조건만으로는 부족합니다. 최대 반복 수, 실패 출구, 사람 승인 게이트까지 같이 둬야 같은 자리를 맴돌지 않습니다.

🔧

승인 게이트

머지, 배포, 삭제처럼 되돌리기 어려운 동작 앞에는 사람이 섭니다.

자동 실행과 최종 결정은 분리
한 바퀴의 기본 흐름

의도 › 맥락 › 실행 › 관찰 › 조정. 이 다섯 단계가 한 바퀴입니다. 목표를 달성하거나 멈출 조건을 만날 때까지 같은 바퀴를 반복합니다. 제안만 하는 AI는 루프가 아닙니다. 실행하고, 결과를 보고, 고쳐서 다시 돌려야 루프입니다.

운영 점검

품질과 비용은 루프를 돌리면서 점검합니다

Trigger와 Stop을 정한 뒤에는 루프가 잘 돌고 있는지 봐야 합니다. 자동으로 돌아간다고 결과가 저절로 좋아지지는 않습니다. 루프는 기존 코드, 문서, 검증 절차, 비용 예산을 그대로 증폭합니다.

품질 관리

  • 코드베이스와 문서 자체를 깨끗하게 유지
  • 검증 절차를 Skill이나 체크리스트로 남김
  • 최신 공식 문서를 쉽게 찾도록 연결
  • 만드는 에이전트와 리뷰 에이전트를 분리

토큰 관리

  • 일에 맞는 루프와 모델을 고름
  • 성공 조건과 멈춤 조건을 구체적으로 둠
  • 큰 실행 전 작은 범위로 시험
  • 결정적 반복 작업은 스크립트로 처리
1
검증은 분리합니다
작업한 에이전트가 스스로 합격을 매기면 자기중심적 평가에 빠집니다. 작성자와 검토자를 나누고, 가능하면 독립된 채점 루브릭을 둡니다.
2
실패를 시스템에 반영합니다
한 번 고치고 끝내지 않습니다. 같은 문제가 다음 바퀴에서 다시 생기지 않도록 규칙, Skill, 체크리스트에 넣습니다.
3
사용량을 봅니다
`/usage`, `/goal`, `/workflows` 같은 사용량 확인 명령으로 어떤 Skill, 서브에이전트, MCP가 비용을 쓰는지 봅니다.
실행법

/goal, /loop, /schedule을 구분해서 씁니다

목표가 분명하면 /goal을 씁니다. 내 컴퓨터에서 반복 감시할 때는 /loop을 씁니다. 클라우드에서 정해진 시각에 돌릴 때는 /schedule 또는 Routines를 씁니다. 프로액티브 루프는 여기에 동적 워크플로우와 자동 모드를 더합니다.

1
완료 기준이 있으면 /goal
점수, 테스트 통과, 체크리스트처럼 끝을 판정할 수 있는 일에 씁니다.
2
정해진 간격으로 확인하면 /loop
로컬 컴퓨터에서 PR, 리뷰, CI처럼 바깥 상태가 바뀌는 일을 주기적으로 봅니다.
3
클라우드에서 돌리면 /schedule 또는 Routines
컴퓨터가 꺼져 있어도 정해진 시각에 실행해야 하는 일은 클라우드 예약으로 옮깁니다.
클라우드로 옮기면 Routines입니다

/loop은 지금 열어 둔 컴퓨터에서 돕니다. 컴퓨터가 꺼지면 멈춥니다. 사람이 없어도 정해진 시각에 클라우드에서 돌려야 한다면 Routines 예약 실행으로 옮깁니다.

루프라는 말의 출발점

“As big as the step from source code to agents was, loops are just as important and as big a step.”

소스 코드에서 에이전트로 넘어간 것만큼, 루프도 그에 못지않게 중요하고 큰 도약입니다. 그는 이제 프롬프트를 일일이 쓰지 않고, 루프가 클로드에게 할 일을 시키게 한다고 말합니다. 이 페이지가 다루는 루프 엔지니어링은 여기서 출발했습니다.
