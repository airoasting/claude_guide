하네스 엔지니어링이란 AI가 일하는 환경 전체를 설계하는 것

한 줄 정의

하네스는 모델을 감싸 일을 시키는 구조 전체입니다. 무엇을 알려주고(맥락), 누가 만들고 채점하며(역할·기준), 어떤 도구를 쥐여줄지를 정해 둔 판. 모델은 그 안에서 돌아가는 엔진 하나일 뿐입니다.

🐎
비유 1: 말과 마구

말의 (근육, 속도)는 AI 모델에 해당합니다.
말에게 씌우는 마구, 달리는 길, 신호 체계는 하네스에 해당합니다.

아무리 좋은 말이라도 올바른 마구 없이는 원하는 방향으로 달리지 못합니다. AI도 마찬가지입니다.

모델 = 말의 근육·속도 하네스 = 마구·길·신호 체계
🧑‍💼
비유 2: 신입사원과 온보딩

아무리 똑똑한 신입사원도 업무 매뉴얼 없이 첫날부터 혼자 일하면 실수투성이입니다.
업무 범위, 보고 체계, 사용 도구, 금지 사항을 정리해 둔
온보딩 시스템 전체가 바로 하네스입니다.
AI에게 "이렇게 일해"라고 알려주는 회사 시스템이 곧 하네스입니다.

모델 = 신입의 머리(지능) 하네스 = 온보딩 시스템 전체

경쟁력은 누가 더 좋은 모델을 쓰느냐가 아니라,
누가 더 좋은 하네스를 설계했느냐에서 나옵니다.

Anthropic이 직접 공개한 하네스 설계 보고서
2026년 3월 24일, Anthropic Labs의 Prithvi Rajasekaran 발표

"장기 실행 앱 개발을 위한 하네스 설계"에서 Anthropic은 하네스 없이 AI에게 긴 작업을 맡겼을 때 반복적으로 나타나는 두 가지 근본 문제를 짚었습니다.

01
자기 평가의 함정 Self-Evaluation Trap

자기 결과물을 스스로 채점하면 항상 후하게 나옵니다

자신이 생성한 작업물을 평가하라고 하면, 에이전트는 자신감 있게 칭찬하는 경향을 보입니다. 인간 관찰자에게는 품질이 명백히 평범한 수준인데도 말입니다.

Prithvi Rajasekaran, Anthropic Labs
일상 비유

학생이 자기 시험지를 직접 채점하면 점수가 후하게 나오는 것과 같습니다.

02
컨텍스트 불안 Context Anxiety

긴 작업 도중에 갑자기 대충 마무리합니다

모델은 긴 작업에서 컨텍스트 윈도우가 차오르면 일관성을 잃습니다. 일부 모델은 "컨텍스트 불안"을 보이며, 한계에 가까워졌다고 인지하면 작업을 조기에 마무리해 버립니다.

Prithvi Rajasekaran, Anthropic Labs
일상 비유

10페이지짜리 보고서를 시켰는데 7페이지에서 갑자기 "이상 마치겠습니다"라고 끝내버리는 셈입니다.

원칙 1
역할 분리 Role·Rubric
↳ 자기 평가의 함정을 해결

"만드는 사람"과 "검토하는 사람"은 반드시 분리해야 합니다. 만드는 AI(Generator·생성자)와 평가하는 AI(Evaluator·채점자)를 따로 두고, 합격 기준(Rubric)을 채점자가 외부에서 쥔 채 냉정하게 채점하게 만듭니다.

원칙 2
컨텍스트 리셋 Context
↳ 컨텍스트 불안을 해결

이전 상태를 "구조화된 MD 파일"로 인수인계해야 합니다. 작업 기억(컨텍스트 윈도우)을 완전히 비우고, 교대 근무자에게 인수인계서를 넘기듯 새 세션을 시작합니다.

만드는 자와 검토하는 자를 분리하고, 이전 맥락은 문서로 넘깁니다.
이 두 원칙이 긴 작업을 끝까지 무너지지 않게 만듭니다.

사례 1: Generator-Evaluator 구조

리서치 브리프 품질 루프

"이 브리프 믿어도 되나?"라는 주관적 질문을, 만드는 AI와 검수하는 AI로 나눠 점수로 바꾼 결과

실습 2에서 만든 자동 브리프를 떠올려 보세요. 한 AI가 기사를 모으고, 요약하고, "우리 사업에 주는 의미"까지 적었습니다. 그런데 그 AI에게 "이 브리프 정확해?"라고 물으면 자기 결과물이라 늘 후하게 답합니다. 바로 자기 평가의 함정입니다. 그래서 만드는 AI(Generator)와 검수하는 AI(Evaluator)를 나눕니다. 검수자는 출처·사실·인사이트의 구체성을 외부 기준으로 냉정하게 채점합니다.

사실 이건 새로운 게 아닙니다

생산 라인과 품질검사를 나누는 것과 같습니다. 만드는 손과 검사하는 손을 분리하는 것입니다. LG전자가 수십 년간 공장에서 해온 일을, 문서 한 장에 그대로 옮긴 것뿐입니다. 사장님은 이미 이 구조의 전문가입니다.

한 번 돌리면 이렇게 바뀝니다
항목1회차 · 검수 전4회차 · 검수 후
출처홍보성 보도자료 혼입tier-1 매체·원문 링크만
사실수치 출처 불명숫자·인용 모두 원문 대조
인사이트"주목된다" 수준"환율 10% 상승 시 로봇 부품 원가 Y" 수준

검수 기준을 외부에서 쥐자, 브리프의 성격 자체가 바뀝니다. 위는 동작 방식을 보여주는 예시입니다.

근거 · Anthropic 실측

Anthropic이 같은 Generator-Evaluator 구조를 측정한 결과, 첫 회차부터 아무 기준 없이 만든 것보다 품질이 뚜렷이 높았습니다. "채점 기준의 문구 자체가 결과물의 성격을 직접 형성한다"는 것이 핵심 발견입니다. (출처는 하단)

어떻게 가능했는가: 두 원칙의 적용

원칙 1 적용

역할 분리

브리프를 쓰는 AI와 채점하는 AI를 완전히 나눈다

  • Generator는 브리프를 쓰고, Evaluator는 출처를 실제로 열어 보고 채점
  • Evaluator는 4가지 기준으로 분해 (출처 신뢰도, 사실 정확성, 인사이트 구체성, 광고·중복 혼입)
  • 최대 5회 반복하며 개선
원칙 2 적용

컨텍스트 리셋

두 AI는 대화하지 않고 파일로만 주고받는다

  • 같은 메시지 창에서 대화하지 않음
  • Generator는 brief.md에, Evaluator는 채점 결과를 review.md에 기록
  • 매 회차 컨텍스트를 처음부터 새로 시작
원칙 1 풀어보기
두 AI는 입력·출력·임무가 정확히 이렇게 다릅니다
Generator역할 1 · 작성자

리서치 정의서에 맞춰 브리프를 쓰고, 지적받은 항목만 다시 고친다

  • 입력리서치 정의서 + 직전 회차의 review.md
  • 출력brief.md (요약 + 출처 + 인사이트)
  • 임무9.5점 미만 항목만 골라 다시 작성
  • 안 함자기 브리프 채점, 점수 변경
Evaluator역할 2 · 검수자

출처를 직접 열어 보고 4가지 기준으로 냉정하게 채점한다

  • 입력brief.md (Generator가 방금 쓴 결과)
  • 출력review.md (점수표 + 개선 사항)
  • 임무출처 신뢰도, 사실 정확성, 인사이트 구체성, 광고 혼입을 각 10점으로 평가
  • 안 함브리프 직접 수정, 다음 회차 작성

프롬프트 예시

간략 버전으로 간단하게 실습을 하고, 상세 버전으로 설계하세요

간략 버전 (한계점: 한 모델이 두 역할을 흉내내 진짜 분리는 아님) 
# 이 프롬프트가 하려는 일
주간 리서치 브리프 한 부를 자동으로 데스크 수준까지 끌어올린다.
같은 AI에게 작성과 검수를 모두 맡기면 자기 결과에 후해지는 문제가 있다.
그래서 두 역할을 나눈다. 쓰는 Generator, 채점하는 Evaluator.

# 등장 인물
Generator (작성자)
  · 입력: 리서치 정의서(키워드·소스·제외 기준) + (있으면) 직전 채점표
  · 출력: 브리프 한 부 (요약 + 출처 링크 + "우리 사업에 주는 의미")
  · 임무: 4가지 채점 항목을 모두 9.5점 이상으로 끌어올리기
  · 안 함: 자기 결과 채점, 점수 임의 변경

Evaluator (검수자)
  · 입력: 방금 Generator가 쓴 브리프
  · 출력: 항목별 점수 + 개선 사항
  · 임무: 출처를 실제로 열어 확인하고 냉정하게 채점, 점수보다 개선 사항을 먼저 적기
  · 안 함: 브리프 직접 수정, 다음 회차 작성

# 합격 기준 (각 10점 만점)
1. 출처 신뢰도: tier-1 매체·원문인가, 홍보성 보도자료는 아닌가
2. 사실 정확성: 숫자와 인용이 원문과 일치하는가
3. 인사이트 구체성: "우리 사업에 주는 의미"가 행동으로 이어지는가
4. 잡음 제거:   광고·중복·출처 불명이 걸러졌는가

# 입력
리서치 정의서: (실습 2에서 만든 정의서를 붙여넣기)

# 진행 순서
한 회차 = Generator 출력 → Evaluator 출력.
2회차부터 Generator는 직전 채점표에서 9.5점 미만 항목만 손본다. 나머지는 절대 건드리지 않는다.

# 종료 조건
4개 항목 모두 9.5점 이상이면 종료. 아니면 다음 회차로. 최대 5회까지.
상세 버전 
# 이 프롬프트가 하려는 일
주간 리서치 브리프를 자동으로 데스크 수준까지 끌어올린다.
간략 버전과 달리, Claude Code의 Task 도구로 Generator와 Evaluator를 별도의 서브에이전트로 띄운다.
두 에이전트는 매 회차 새 컨텍스트에서 시작하고, 같은 메시지 창에서 대화하지 않고 파일로만 주고받는다.

# 작업 디렉토리
./brief-loop/
  ├─ brief.md    (Generator의 출력)
  └─ review.md   (Evaluator의 출력)

# 합격 기준 (각 10점 만점, 총 40점)
1. 출처 신뢰도:  tier-1 매체·원문인가, 홍보성 보도자료는 아닌가
2. 사실 정확성:  숫자·인용이 원문과 일치하는가
3. 인사이트 구체성: "우리 사업에 주는 의미"가 행동으로 이어지는가
4. 잡음 제거:    광고·중복·출처 불명이 걸러졌는가

# 입력
리서치 정의서: (실습 2에서 만든 키워드·소스·제외 기준)

# 메인 에이전트 절차 (최대 5회)
회차마다 다음 3단계를 순서대로 수행한다.

1. Task 도구로 generator-agent를 호출한다.
   입력: 리서치 정의서 + (있으면) ./brief-loop/review.md
   산출물: ./brief-loop/brief.md

2. Task 도구로 evaluator-agent를 호출한다.
   입력: ./brief-loop/brief.md
   산출물: ./brief-loop/review.md

3. review.md를 읽는다.
   모든 항목이 9.5점 이상이면 STATUS: COMPLETED를 선언하고 종료.
   아니면 다음 회차로 넘어간다.

# generator-agent 시스템 프롬프트
너는 주간 리서치 브리프를 쓰는 Generator다.
- 리서치 정의서를 읽는다. 직전 review.md가 있으면 함께 읽는다.
- 4가지 합격 기준을 목표로 브리프를 ./brief-loop/brief.md에 저장한다.
- 모든 주장에 원문 출처 링크를 단다. 직전 review.md가 있으면 9.5점 미만 항목만 수정한다.
- 수정한 항목에는 다음 표시를 단다.
  # 수정 (N회차): [항목명] - [변경 내용]

# evaluator-agent 시스템 프롬프트
너는 브리프를 검수하는 Evaluator다.
- ./brief-loop/brief.md를 읽고, 인용된 출처를 실제로 열어 확인한다.
- 4가지 항목을 각 10점 만점으로 채점하고, 결과를 ./brief-loop/review.md에 다음 형식으로 기록한다.

  | 회차 | 출처 신뢰도 | 사실 정확성 | 인사이트 구체성 | 잡음 제거 | 총점 |
  |  N   |    X점     |    X점     |      X점       |   X점    | XX점 |

  표 다음에 9.5점 미만 항목별로 "무엇이 부족한가, 무엇을 바꿔야 하는가"를 구체적으로 적는다.
- 9점 이상은 출처와 사실이 모두 확인될 때만 준다. 점수보다 개선 사항을 먼저 적는다.
- 브리프를 직접 수정하지 않는다.
사례 2: Planner-Generator-Evaluator 구조

경영 시뮬레이션 의사결정 루프

"AI는 어디까지, 사람은 어디서 승인하는가"를 plan.md 한 장에 못박은 결과

실습 3에서 환율·원자재 변동이 원가와 KPI에 주는 영향을 시뮬레이션했습니다. 거기서 진짜 중요한 건 시뮬레이터가 아니라 경계였습니다. 계산은 AI가, 가정의 승인과 의사결정은 사람이 합니다. 그 경계를 구조로 굳히면 3-Agent 하네스가 됩니다. 사람(사장님)이 Planner 단계에서 합격 기준과 승인할 가정을 plan.md에 못박고, 그 기준은 Generator도 Evaluator도 바꾸지 못합니다. plan.md를 쥔 사람이 의사결정 권한을 쥡니다.

이것도 새로운 게 아닙니다

사업계획을 세울 때 먼저 목표 마진·리스크 한계를 합의하고(기획), 실무가 시나리오를 돌리고(실행), 재무·리스크가 검증하는(감사) 순서 그대로입니다. 다른 점은 단 하나. 합의문이 사람 머릿속이 아니라 plan.md라는 문서에 박혀, 누구도 임의로 못 바꾼다는 것입니다.

plan.md 한 장이 경계를 만듭니다
plan.md에 사람이 못박는 것그래서 AI가 못 하는 것
합격 기준 · "영업이익률 X% 이하면 FAIL"기준을 느슨하게 바꿔 통과시키기
승인 가정 · "환율·원자재 가정 변경은 사람 승인 필수"가정을 몰래 바꿔 결과 미화하기
판단 경계 · "투자 결정은 사람, 계산·제안은 AI"스스로 의사결정을 내리기
근거 · Anthropic 실측

Anthropic이 같은 3-Agent 구조로 풀스택 앱을 만든 실측에서, 반복할수록 시간과 비용이 급격히 줄었습니다. 3차 빌드는 1차의 약 1/12로 끝났습니다. Evaluator가 남은 문제만 정확히 짚어 주기 때문입니다. (출처는 하단)

어떻게 가능했는가: 두 원칙의 적용 (3-Agent로 확장)

전사 의사결정은 브리프보다 한 단계 더 무겁습니다. 사례 1의 2-Agent 구조에 Planner(기획자)를 더하고, 시작 전에 "무엇이 성공이고 어디까지가 사람 몫인지"를 먼저 합의하는 스프린트 계약을 둡니다.

원칙 1 적용

역할 분리

Planner를 더해 기획·계산·검증을 모두 분리한다

  • Planner가 합격 기준과 판단 경계를 plan.md로 확정
  • Generator가 시나리오 계산 모델을 짜고, Evaluator가 결과를 기준으로 판정
  • Evaluator는 PASS/FAIL만 판정한다 (가정·기준은 못 바꿈)
원칙 2 적용

컨텍스트 리셋

합격 기준을 먼저 합의하고 모든 소통은 파일로 한다

  • 스프린트 계약: 시작 전에 plan.md로 합격 기준 합의 ("좋아 보인다" 대신 측정 가능한 조건)
  • Generator는 계산 결과를, Evaluator는 판정을 feedback.md로 남김
  • 매 회차 컨텍스트를 새로 시작
원칙 1 풀어보기
세 역할의 분담: 누가 무엇을 하고, 사람은 어디서 승인하나
Planner역할 1 · 기획자 (사람이 승인)

시작 전에 합격 기준과 판단 경계를 사람과 합의한다

  • 입력1~4문장의 시뮬레이션 요청
  • 출력plan.md (합격 기준 + 승인할 가정 + 판단 경계)
  • 임무"좋아 보인다" 대신 PASS/FAIL 기준 + 사람 승인 지점 명시
  • 안 함계산, 합격 판정
Generator역할 2 · 제작자

기준대로 시나리오를 계산하고, FAIL 항목만 고친다

  • 입력plan.md + 직전 회차의 feedback.md
  • 출력시나리오별 원가·KPI 계산 결과
  • 임무FAIL 기준을 PASS로 바꾸는 데 집중
  • 안 함합격 기준·가정 변경, 자기 판정
Evaluator역할 3 · 검증자

계산을 검증하고 plan.md 기준으로 PASS/FAIL을 판정한다

  • 입력Generator가 방금 낸 계산 결과
  • 출력feedback.md (기준별 PASS/FAIL + 근거)
  • 임무계산 검증, 가정 위반·리스크 임계 초과 표시
  • 안 함결과 수정, 합격 기준 변경

프롬프트 예시

간략 버전으로 간단하게 실습을 하고, 상세 버전으로 설계하세요

간략 버전 (한계점: 한 모델이 세 역할을 흉내내 진짜 분리는 아님) 
# 이 프롬프트가 하려는 일
환율·원자재 변동이 원가와 KPI에 주는 영향을 시뮬레이션한다.
"좋아 보인다"라는 주관에서 벗어나 PASS/FAIL로 판정 가능한 합격 기준을 먼저 합의하고,
사람이 승인할 가정과 판단 경계를 못박은 뒤에 계산을 반복한다.

# 등장 인물
Planner (기획자, 사람이 승인)
  · 입력: 1~4문장의 시뮬레이션 요청
  · 출력: 합격 기준 + 승인할 가정 + 판단 경계가 적힌 plan.md
  · 임무: "좋아 보인다" 대신 PASS/FAIL 기준과 사람 승인 지점 설계
  · 안 함: 계산, 합격 판정

Generator (제작자)
  · 입력: plan.md + (있으면) 직전 회차 피드백
  · 출력: 시나리오별 원가·KPI 계산 결과
  · 임무: FAIL 기준을 PASS로 바꾸기
  · 안 함: 합격 기준·가정 변경, 자기 판정

Evaluator (검증자)
  · 입력: 방금 Generator가 낸 결과
  · 출력: 기준별 PASS/FAIL + 근거
  · 임무: 계산을 검증하고 가정 위반·리스크 임계 초과를 표시
  · 안 함: 결과 수정, 합격 기준 변경

# 입력
시뮬레이션 요청: (예: 환율 USD +10% 시 가전·로봇 주력 제품 원가·이익률 영향)

# 진행 순서 (최대 3회차)
1회차에만 Planner를 호출한다.
  · Planner: 측정 가능한 합격 기준과 사람 승인 지점을 plan.md로. 주관적 표현 금지.
  · 사람이 plan.md를 검토·승인한 뒤에만 다음으로 넘어간다.
  · Generator: plan.md대로 시나리오를 계산.
  · Evaluator: 기준별 결과를 | 기준 | 판정 | 근거 | 표로 출력. 점수보다 근거를 먼저 적는다.

2회차부터:
  · Generator: 직전 회차에서 FAIL인 기준만 손본다. 나머지는 그대로 둔다.
  · Evaluator: 다시 판정한다.

# 종료 조건
모든 기준 PASS면 종료. 아니면 다음 회차로. 최대 3회까지.
상세 버전 
# 이 프롬프트가 하려는 일
경영 시뮬레이션을 PASS/FAIL 기반으로 자동 구축한다.
"이 판단이 안전한가"를 측정 가능한 합격 기준과 사람 승인 지점으로 바꾼 뒤,
Claude Code의 Task 도구로 Planner / Generator / Evaluator를 별도 서브에이전트로 띄운다.
세 에이전트는 매 단계 새 컨텍스트에서 시작하고, 같은 메시지 창에서 대화하지 않고 파일로만 주고받는다.

# 작업 디렉토리
./sim-build/
  ├─ plan.md       (Planner의 합격 기준 + 승인 가정 + 판단 경계)
  ├─ model/        (Generator의 계산 모델·결과)
  └─ feedback.md   (Evaluator의 PASS/FAIL + 근거)

# 입력
시뮬레이션 요청: (한두 문장. 예: 환율 +10% 시 주력 제품 원가·이익률 영향)

# Phase 1. 기획·합의 (사람 승인 필수)
계산을 시작하기 전에 합격 기준과 경계를 먼저 굳힌다.

1. Task 도구로 planner-agent를 호출한다.
   입력: 시뮬레이션 요청
   산출물: ./sim-build/plan.md

2. 사람(의사결정자)이 plan.md를 검토한다.
   합격 기준·가정·판단 경계가 타당하면 plan.md 하단에 STATUS: APPROVED 추가.
   아니면 1번으로 돌아가 기준을 다시 쓴다. 이 승인은 사람만 한다.

# Phase 2. 계산·검증 (최대 N회차)
기준이 굳어지면 그제서야 시나리오를 돌린다.

회차마다 다음 3단계를 순서대로 수행한다.

1. Task 도구로 generator-agent를 호출한다.
   입력: plan.md + (있으면) feedback.md
   산출물: ./sim-build/model/

2. Task 도구로 evaluator-agent를 호출한다.
   입력: plan.md + ./sim-build/model/
   산출물: ./sim-build/feedback.md

3. feedback.md를 읽는다.
   모든 기준이 PASS면 STATUS: COMPLETED를 추가하고 종료.
   아니면 다음 회차로.

# planner-agent 시스템 프롬프트
너는 시뮬레이션 기획자다. 1~4문장 요청을 받아 plan.md를 작성한다.
- 합격 기준은 번호가 매겨진 측정 가능한 조건 5개 이상. (예: "환율 +10% 시 어떤 제품도 영업이익률 X% 미만이면 FAIL")
- 사람이 승인해야 하는 가정과, AI가 못 넘는 판단 경계를 명시한다.
- 주관적 표현 금지. 데이터 출처·기준일·단위도 함께 명시한다.

# generator-agent 시스템 프롬프트
너는 계산 모델 작성자다. plan.md만을 진실의 원천으로 보고 ./sim-build/model/에 시나리오를 계산한다.
- (있으면) 직전 feedback.md에서 FAIL인 기준 번호만 수정한다. 나머지는 절대 건드리지 않는다.
- plan.md의 가정을 임의로 바꾸지 않는다. 가정 변경이 필요하면 멈추고 사람 승인을 요청한다.

# evaluator-agent 시스템 프롬프트
너는 검증자다. plan.md의 합격 기준만을 근거로 ./sim-build/model/의 결과를 판정한다.
- 계산을 다시 짚어 검증하고, 가정 위반·리스크 임계 초과가 있으면 표시한다.
- 결과를 feedback.md에 다음 형식으로 기록한다.

  | 기준 번호 | 판정 (PASS/FAIL) | 근거 |

- 근거는 재현 가능하게 구체적으로 적는다.
- 결과를 직접 수정하지 않는다.
