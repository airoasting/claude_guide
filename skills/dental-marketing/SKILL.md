# BrandLab 치과 브랜드 전략 벤치마킹 + 제안서 자동화 스킬

지정 지역 치과들의 온라인 홍보 채널과 전략을 자동으로 수집·분석해 보고서화하고, 클라이언트에게 바로 전달 가능한 BrandLab 브랜드 전략 제안서까지 5color 품질 루프를 거쳐 생성한다.

## 언제 쓰는가 / 안 쓰는가

**쓴다:** 지역 + 과목(기본 치과) + 채널 벤치마킹 또는 제안서 생성 의도가 함께 있을 때.
**안 쓴다:** 기존 제안서 단순 평가·요약·번역만 원할 때, 의료와 무관한 시장조사, 단일 병원 1건 확인 요청.

**BrandLab 포지셔닝:** "환자가 선택하는 병원을 만듭니다 — 브랜드 전략부터 수익 구조까지". 광고대행사는 트래픽을 만들고, BrandLab은 선택받는 이유를 만든다.

## 산출물

세 개의 docx를 생성해 워크스페이스 폴더에 저장한다.

| # | 파일명 | 내용 | 생성 스크립트 |
|---|--------|------|--------------|
| 1 | `치과_홍보채널_조사표.docx` | 병원별 링크 목록 표 (홈페이지·블로그·플레이스·SNS) | `scripts/make_link_table.js` |
| 2 | `치과_홍보대행_벤치마킹_분석보고서.docx` | 내부 전략 분석 (전략 9가지 패턴 + 원장 브랜딩·AEO·수익 구조 + BrandLab 부가서비스 10종 ①~⑩) | `scripts/make_analysis.js` |
| 3 | `BrandLab_브랜드전략_제안서.docx` | 클라이언트용 제안서 (5color 합격본) | `scripts/make_proposal.js` |

각 docx의 섹션 구조와 작성 원칙은 `references/document-specs.md`에 있다.

## 실행 워크플로우

### STEP 0 — 사전 준비 (1회)
스크립트는 Node.js `docx` 라이브러리를 쓴다. 스킬 폴더에서 1회 설치한다.
```bash
cd [스킬 폴더] && npm install docx
```

### STEP 1 — 조사 대상 확정
대화에 없으면 사용자에게 확인한다(있으면 건너뜀).
- **지역**: 조사할 지역구·시·동 (예: "강남구", "강남구+수지구")
- **병원 수**: 지역당 몇 곳 (기본값 15곳)
- **저장 폴더(`OUT_DIR`)**: 결과 docx 3종을 저장할 폴더. 미지정 시 스크립트 실행 시점의 현재 작업 디렉터리(`process.cwd()`)에 저장된다. 세 스크립트 모두 `node scripts/<파일>.js [저장 폴더]` 형태로 첫 인자에 폴더를 넘기거나 `OUT_DIR` 환경변수로 지정할 수 있고, 폴더가 없으면 자동 생성한다. 세 산출물을 같은 폴더에 모으려면 STEP 3·5·6에서 동일한 폴더를 넘긴다.

### STEP 2 — 병원 목록 수집
WebSearch로 각 지역 치과를 수집한다. 검색어·보조 소스·채널별 링크 수집법·접근 제한 우회는 **`references/research-protocol.md` §1**을 따른다.

### STEP 3 — 1차 산출물: 링크 조사표
`node scripts/make_link_table.js [저장 폴더]`로 `치과_홍보채널_조사표.docx`를 생성한다(폴더 생략 시 현재 작업 디렉터리). 구성: `references/document-specs.md` §1.

### STEP 4 — 홍보 전략 심층 조사
팔로워 상위 병원의 홈페이지·인스타그램을 방문해 4개 항목군(기본 채널 전략 / 원장 브랜딩 / AEO / 수익 구조)을 기록한다. 항목 체크리스트는 **`references/research-protocol.md` §2**.

### STEP 5 — 2차 산출물: 벤치마킹 분석 보고서
`node scripts/make_analysis.js [저장 폴더]`로 `치과_홍보대행_벤치마킹_분석보고서.docx`를 생성한다(STEP 3과 같은 폴더를 넘긴다). 구성: `references/document-specs.md` §2.

### STEP 6 — 3차 산출물: 제안서 초안
`node scripts/make_proposal.js [저장 폴더]`로 `BrandLab_브랜드전략_제안서.docx` 초안을 생성한다(STEP 3·5와 같은 폴더). 구성과 **작성 원칙(5color 합격의 사전 조건)**은 `references/document-specs.md` §3에 있다. 초안 단계에서 그 원칙을 미리 지킬수록 STEP 7 라운드 수가 줄어든다.

### STEP 7 — 5color 반복 개선 루프 (최대 10회)
제안서(파일 3번)만 대상으로 5-Color Harness 평가→수정→재생성을 반복한다. 합격선 **RED·SILVER·BLUE 평균 ≥ 9.5 AND GOLD ≥ 9.5**에 도달하거나 10회에 도달하면 종료한다. 평가 기준표, 자동 감점 항목, 라운드 진행 방식, 점수 추적표, 정체·오류 시 중단 조건은 모두 **`references/5color-loop.md`**에 있다 — 루프를 시작하기 전에 그 문서를 읽는다.

### STEP 8 — 산출물 확인
STEP 3·5·6에서 같은 저장 폴더를 넘겼다면 3개 docx가 이미 그 폴더에 있다. 존재를 확인한다.
```bash
ls -la [저장 폴더]/치과_홍보채널_조사표.docx \
       [저장 폴더]/치과_홍보대행_벤치마킹_분석보고서.docx \
       [저장 폴더]/BrandLab_브랜드전략_제안서.docx
```
다른 폴더에 만들어 워크스페이스로 옮겨야 한다면 `cp [원본 폴더]/*.docx [워크스페이스]/`로 복사한다.

## 기술 스택
- **문서 생성**: Node.js `docx` 라이브러리 (`npm install docx`)
- **웹 조사**: WebSearch + Claude in Chrome(홈페이지 방문)
- **품질 검토**: 5-Color Harness 루프 (RED·SILVER·BLUE·GOLD, 최대 10라운드)
- **저장**: Bash로 워크스페이스 복사

## 엣지 케이스 & 중단 조건
- **인스타그램/네이버 블로그 접근 제한**: 우회법은 `references/research-protocol.md` §1 참고. 끝내 못 구한 링크는 표에 "미확인"으로 표시하고 2차 수작업 가이드를 보고서에 넣는다.
- **스크립트 수정은 Python으로**: make_*.js의 한국어 텍스트를 고칠 때 Edit 도구는 UTF-8 멀티바이트 손상 위험이 있어 금지. 안전한 방법(read→replace→write utf-8 스니펫, 파일 전체 재작성 템플릿, 수정 후 검증)은 `references/customization.md` §0에 있다 — 그 절차를 그대로 따른다.
- **재생성 실패**: `node make_proposal.js`가 오류면 직전 정상 버전으로 롤백하고 사용자에게 보고한다. 깨진 docx를 합격 처리하지 않는다.
- **5color 정체/오류 중단**: 점수 정체·하락·재생성 오류 시 처리는 `references/5color-loop.md` §6.
- **의료광고법**: 과장 표현 병원은 벤치마킹 보고서에 주의 표시. 제안서는 신뢰 기반·비과장 원칙을 명시한다.
- **고정값**: BrandLab 연락처 `paulyoo95@gmail.com`는 제안서에 고정 포함.
