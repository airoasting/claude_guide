# F&B 프랜차이즈 본사 매뉴얼 디자인 스펙 v1.0

> 본 문서는 운영기획본부장 에이전트가 `reference/` 폴더의 17개 완성본 매뉴얼(.docx)을
> 분석하여 정식 수립한 검증 기준입니다.
> SOP 라이터는 본 문서를 100% 따라야 하며, QSC 검수관는 본 문서의 각 항목을
> 측정 가능한 기준으로만 평가합니다.

---

## 0. 분석 근거

`reference/` 폴더의 17개 .docx 매뉴얼 전수 분석 결과 (2026-05-07 추출):

| 패턴 | 검출 빈도 | 적용률 |
|---|---|---|
| `제 N 조 (조항)` 3개 (목적/개요·운영원칙/구성) | 51건 / 17파일 (각 3건) | 100% |
| `STEP N` 단계 표기 (5~7단계) | 81건 / 16파일 | 94% |
| `①②③…` 원문자 하위 항목 | 173건 / 17파일 | 100% |
| `※` 주의·예외 표기 | 92건 / 17파일 | 100% |
| 추상 표현 (`잘`, `원활히`, `적절히` 등) | 0건 (목적문 외) | 0% |

→ **참조 매뉴얼의 구조와 톤은 매우 일관됨.** 본 스펙은 이 패턴을 강제한다.

## 1. 매뉴얼 헤더 (필수)

각 챕터 파일 첫머리에 다음 4개 요소를 순서대로 배치한다.

### 1.1 1행: 상위 매뉴얼명
- 형식: `[부서명] 업무 매뉴얼` 또는 `[부서명] [기능명] 매뉴얼`
- 예: `영업팀 업무 매뉴얼`, `영업팀 배달 플랫폼 관리 매뉴얼`

### 1.2 2행: 챕터 제목 (h1)
- 본 챕터의 작업명을 그대로 (feature_list의 `title` 필드 확장형)
- 예: `해산물 납품 품목 신규 입고 절차`, `고객불만 업무 처리 매뉴얼`

### 1.3 3행: 부제 / 목적 한 줄 요약
- 1~2줄, 챕터의 결과를 현장 언어로
- 예: `가맹점 납품 해산물 품목의 신규 입고 등록 절차를 명확히 하여 관리 일관성을 확보하고 안정적인 식자재 공급을 도모함`

### 1.4 4행: 메타 라인
- 형식: `소관부서 | [팀명]    분류 | [업무 카테고리]`
- 예: `소관부서 | 영업팀    분류 | 물류 관리`

## 2. 본문 구조 (필수 7요소, 순서 고정)

> 아래 7요소 중 **1개라도 누락 시 QSC 검수관 즉시 fail**.

### 2.1 [매뉴얼 목적] 섹션 헤더

별도 헤더 박스 (마크다운에서는 `## [매뉴얼 목적]` 또는 `---`로 구분).

#### 2.1.1 제 1 조 (목적)
- 1단락 (3~6줄)
- 도입 문구: `본 매뉴얼은 …을 목적으로 한다.` 또는 `…하는 데 목적이 있다.`
- 챕터가 무엇을 표준화/관리/규정하는지 명시

#### 2.1.2 제 2 조 (개요·운영원칙)
- 호 단위 5개 이상 (필수 5호 + 필요 시 6호)
- 필수 5호 항목 (순서 고정):
  1. **1호. 주기**: 수행 빈도·트리거 (`수시`, `월 1회`, `사유 발생 시` 등)
  2. **2호. 대상**: 적용 범위 (`전 가맹점`, `이익률 5% 미만 매장` 등)
  3. **3호. 담당자**: 직책·역할 명시 (`물류 담당 실무자`, `슈퍼바이저`, `영업팀 대리` 등)
  4. **4호. 항목**: 다루는 데이터·결과물 (`품목명, 규격, 단가…`)
  5. **5호. 보고방식**: 결재 라인 또는 기록 방식 (`담당자 → 팀장 → 본부장 보고`)
- 필요 시 **6호. 처리 시한** 추가 가능 (`D+1 1차 회신, D+2 최종 회신` 등)

#### 2.1.3 제 3 조 (구성)
- 1줄: `본 매뉴얼은 총 N단계 업무 흐름(① … → ② … → … → ⑤ …)으로 구성된다.`
- N = 5~7 권장 (참조 매뉴얼 평균 5.4단계)

### 2.2 [업무 순서도] 섹션 헤더

#### 2.2.1 STEP 흐름 표 (1행 N열)
- 표 형식, 각 칸: `STEP N\n[단계명]`

#### 2.2.2 STEP 요약 박스 (각 STEP 1개씩, 총 N개)
- 표 형식 (1행 2열): `| STEP N | 단계명 |` + `| | 핵심 액션 3개 |`
- 핵심 액션 3개는 동사형 종결의 짧은 명사구

### 2.3 [업무 상세작성] 섹션 헤더

#### 2.3.1 STEP별 상세 (총 N개 STEP)

각 STEP는 다음 구조:

```
STEP N  [단계명]
① [하위항목명]
[불릿 1: 동사 종결, 구체 행동]
[불릿 2]
[불릿 3]
※ [주의/예외 사항]
② [하위항목명]
...
```

**필수 요소** (각 STEP당):
- ① ② 원문자 하위 항목 **2개 이상**
- 각 ① ② 아래에 불릿 **2~4줄** (동사형, 구체 행동)
- `※` 주의 표기 **1개 이상** (전체 챕터 기준 6개 이상)

### 2.4 [부록] 섹션 (선택, 1개 이상 권장)

복잡한 분류·기준이 있을 경우 표 형식 부록 추가.

예시 형식:
- `[부록 1. 이물질 응대 요약]` (구분 / 예시 / 회수 방식)
- `[부록 2. 보상 전결권]` (보상 금액 / 전결권자 / 비고)
- `[부록 3. 귀책사유 구분]` (구분 / 예시 / 대응 원칙)

### 2.5 [체크리스트] 섹션 헤더

#### 2.5.1 CHECK LIST 헤더 행
- 헤더 라인: `CHECK LIST`
- 작성자 정보 표 (1행): `| 방문일자 | | 담당자 | |`

#### 2.5.2 점검 항목 표 (필수 4컬럼)
- 컬럼 고정: `| 구분 | 순번 | 점검 항목 | 완료 |`
- 점검 항목 **10개 이상** (참조 매뉴얼 평균 11.4개)
- `구분` 컬럼은 그룹별 묶음 (3~5개 그룹 권장, 같은 그룹 셀은 빈칸 처리)
- 점검 항목 종결: `~하였는가`, `~ 완료`, `~ 확인` 중 하나
- `완료` 컬럼: `□` 박스

## 3. 톤 규칙

### 3.1 금지 표현 (QSC 검수관가 grep 검출)

다음 표현이 챕터 본문에 **3건 이상** 발견되면 fail.
(단, 제1조 목적문 1단락 내 1회는 허용 — 참조 매뉴얼에도 `신속하고 정확하게` 형태로 존재)

| 금지 표현 | 사유 | 대체 |
|---|---|---|
| `잘 ` (잘 관리, 잘 처리) | 모호 | 구체 행동·체크포인트 |
| `원활히`, `원활하게` | 모호 | 절차·시간 명시 |
| `적절히`, `적절하게` | 모호 | 기준값·임계값 명시 |
| `꼼꼼히`, `꼼꼼하게` | 모호 | 체크 항목 명시 |
| `대체로`, `일반적으로` | 모호 | 빈도·조건 명시 |
| `필요에 따라` | 모호 | 트리거 조건 명시 |
| `가급적`, `최대한` | 모호 | 임계값 또는 우선순위 명시 |
| `신속하게` | 모호 (목적문 외) | 분 단위 시간 명시 |

### 3.2 권장 표현 (현장 SOP 톤)

- **시간**: `24시간 이내`, `D+1`, `2영업일 전`, `오전 9시까지`
- **수량·임계값**: `±5% 이내`, `5만원 이하`, `15% 이내`, `재고 안전 15% 확보`
- **빈도**: `월 1회 정기 방문`, `주 1회 원격`, `매일 오전`, `분기 1회`
- **금액·결재**: `5만원 이하 담당자 결재 / 30만원 초과 대표이사 결재`
- **수치 KPI**: `원가율 33%`, `이익률 5% 미만`, `폐기율 3% 이내`

### 3.3 어조

- 종결: 명사형 또는 평서형 단언 (`~한다`, `~확인`, `~ 보고`).
- 경어체(`~하시기 바랍니다`) 금지 — 매뉴얼은 본사 내부 SOP.
- 1인칭(`우리`, `저희`) 사용 금지.
- 영문 표기는 원어 그대로 (`POS`, `SV`, `QSC`, `KPI` 등 약어 허용, 첫 등장 시 풀이 권장).

## 4. 실전성 기준 (필수)

### 4.1 STEP별 4요소 (모든 STEP 필수)

각 STEP에는 다음 4가지가 명시되어야 한다.

| 항목 | 기준 | 예시 |
|---|---|---|
| 담당자 | 직책 또는 역할명 | `슈퍼바이저`, `영업팀 대리`, `물류 담당자` |
| 트리거·시점 | 언제 시작하는지 | `통보 수신 직후`, `D+1 09:00 이전` |
| 도구·시스템 | 사용하는 시스템·서식 | `본사 ERP [설정관리]→[매입단가관리]`, `POS 시스템`, `카카오톡 단체방` |
| 체크포인트 | 무엇을 확인하면 끝인지 | `리스트 조회로 등록 여부 확인`, `스크린샷 보관` |

> 4요소는 STEP 본문(불릿) 또는 STEP 요약 박스 어디에든 배치 가능.
> **1개 STEP에서 3개 이상 누락 시 fail.**

### 4.2 수치·임계값 명시

전체 챕터 본문에 다음 중 **3개 이상** 등장해야 한다 (참조 매뉴얼 평균 8개):

- 시간 임계값 (예: `24시간 이내`, `D+2`)
- 비율·금액 임계값 (예: `±5%`, `5만원 이하`)
- 빈도 (예: `월 1회`, `주 3회`)
- KPI 기준 (예: `원가율 33%`, `이익률 5%`)

### 4.3 결재·보고 라인 명시

`제2조 5호. 보고방식`에 결재 라인이 1개 이상 표기되어야 한다.
형식: `담당자 → 팀장 → 본부장` 또는 등가 표현.

### 4.4 ※ 주의 표기 빈도

- 챕터 전체에 `※`로 시작하는 주의·예외 라인이 **6건 이상** (참조 매뉴얼 평균 5.4건).
- 각 ※ 라인은 1줄로 끝나는 단언형.

## 5. 디자인 호환성 (매뉴얼 v4 빌더 변환 가능)

### 5.1 마크다운 포맷 규칙

- 표 문법 정상: `|` 정렬, 헤더 구분선(`---`) 정확.
- 헤딩 레벨: h1 1개 (챕터 제목) → h2 다수 (섹션 헤더 `[매뉴얼 목적]` 등) → h3 다수 (`STEP N`, `제 N 조`).
- 이미지 경로는 상대 경로 (`./assets/...`).
- 코드 블록은 ``` 펜스 사용.
- 깨진 문자(`�`) 금지.

### 5.2 PPTX 변환 시 매핑 (참고)

| 마크다운 요소 | PPTX 변환 결과 |
|---|---|
| `## [매뉴얼 목적]` | 좌측 빨간 밴드 + 섹션 헤더 슬라이드 |
| `### 제 N 조 (...)` | 노란 박스 헤더 |
| STEP 흐름 표 | 가로 화살표 SmartArt |
| STEP 요약 박스 | 카드형 박스 (3개 액션 표시) |
| `※` 라인 | 회색 작은 글씨 캡션 |
| CHECK LIST 표 | 체크박스 표 슬라이드 |

### 5.3 색상·타이포 (PPTX 빌더 자동 적용, 작성자 신경 X)

- 좌측 빨간 밴드: `#B91C1C`, 폭 페이지의 약 8%
- 노란 박스: `#FEF3C7`
- 한글 폰트: `+mj-ea`
- 영문 폰트: 시스템 기본 sans-serif
- line-height: 150%
- 제목 20pt / 본문 13pt / 캡션 11pt

## 6. 평가 게이트 (QSC 검수관 체크 매트릭스)

각 챕터는 아래 4개 카테고리 모두 pass여야 done 처리.

### A. 구조 검증 (필수 7요소)
- [ ] 헤더 4행 (상위명/제목/부제/메타) 모두 존재
- [ ] `[매뉴얼 목적]` + 제1조·제2조·제3조 모두 존재
- [ ] 제2조에 5호 이상 존재
- [ ] `[업무 순서도]` + STEP 흐름 표 + STEP 요약 박스 N개 존재
- [ ] `[업무 상세작성]` + 모든 STEP에 ①② 하위 항목 존재
- [ ] `[체크리스트]` + 4컬럼 표 + 점검 항목 10개 이상
- [ ] STEP 수: 5~7 범위

> 1개라도 누락 시 fail.

### B. 톤 검증
- 금지 표현 (3.1) 본문 누적 발견 ≥ 3 → fail.
- ※ 주의 라인 < 6건 → fail.

### C. 실전성 검증
- 4.1의 4요소 (담당자/트리거/도구/체크포인트) 중 3개 이상 누락된 STEP 1개라도 발견 → fail.
- 4.2의 수치·임계값 < 3건 → fail.
- 4.3의 결재라인 미명시 → fail.

### D. 디자인 호환성 검증
- 표 문법 깨짐, 헤딩 레벨 오류, 깨진 문자 발견 → fail.

### E. fail 사유 기록 형식 (QSC 검수관가 작성)

`shared/eval_report.json`에 다음 형식으로:
```
"fail_reasons": [
  "L23: 제2조 5호(보고방식) 누락",
  "L67: '잘 관리한다' 1건 → 구체 행동 명시",
  "L92: STEP 3에 담당자·도구·체크포인트 모두 누락"
]
```
